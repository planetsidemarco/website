Rocketry

During my Honours final year at the University of Canterbury, I took the ENGR402 Rocketry course and had an amazing time as Avionics and GNC Team Lead.

For this project, we worked in a small team of 8 students to design, build, and program a 100 mm rocket to be flown with canard roll-control.

As Team Lead, I oversaw the design, manufacturing, and assembly of the electronic stack, along with the programming and testing of the roll-control software required for controlled flight.

This project required skills such as CAD design, programming, electrical soldering, component assembly, and workshop machining.

Avionics Stack

When beginning the project, we were tasked with designing an electronic stack layout with 3D-printed parts. The required parts to be considered in the design were:

  • Flight Controller PCB (with XBee)
  • 2 Cell LiPo, Telemetrum, and Manual Deployment batteries
  • Power switch
  • Telemetrum
  • 900MHz and 433MHz Antennae
  • External GPS Board
  • Manual Deployment PCB
  • Dragon Link receiver
  • Associated wiring

We succeeded in the designing, manufacturing, and assembly of the avionics stack as shown, directly mounted to the actuator stack.

Activating Launch Controller

Directly after the rocket launches, the launch controller is expected to turn on once escaping the guide rails. The activation requirements are:

  • Distance calculated by IMU double integration > 6 m.

OR

  • Launch time = T + 1 seconds.

For this system, an important aspect to consider is IMU integration drift, due to the use of IMU double integration for determining distance which causes significant drift over time without appropriate sensor fusion.

This issue with drift is solved by setting the acceleration variable to zero until launch is detected when IMU acceleration > 0.5 ms-2.

Detecting Apogee & Restarting Data Log

In the event of a failed parachute deployment and crash, there is a chance of a data log failure, resulting in a loss of data.

To ensure we did not lose all flight data in the event of a crash landing, we chose to detect apogee and restart the data log to save the launch data to apogee.

This was achieved by stopping the initial data log at apogee and starting a new log afterwards. This ensures that initial flight data leading up to apogee is saved to the SD card so that in the event of a crash, only the flight data between apogee and crashing may be lost.

The first was the 'Apogee Check Flag'.

  • A max altitude variable is continuously updated along with a previous max altitude, separated by a 1 second delay.
  • If both variables are equal, apogee check flag is set to true.

The second was the 'Apogee Confirmed Flag'.

  • A max altitude variable is continuously updated along with a previous max altitude, separated by a 1 second delay.
  • If both variables are equal, as well as the apogee check flag set to true, apogee confirmed flag is set to true.
  • If not equal, apogee check flag is reset to false.

After both of these flags becomming set, the final flag set was the 'Apogee Reached'.

  • Once both flags are set to true, apogee has been reached.
